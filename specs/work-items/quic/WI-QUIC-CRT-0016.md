REQ-QUIC-CRT-0103 - REQ-QUIC-CRT-0118 - REQ-QUIC-RFC9001-S8-0002  ## Design Inputs  - ARC-QUIC-CRT-0016  ## Planned Changes  - Add a narrow server-role branch to the existing peer transport-parameter commit gate so commit can occur only after the inbound client Finished proof exists, the staged peer transport parameters still match the candidate parameters, the completed Finished transcript metadata still describes that TLS session, and the bridge is non-terminal. - Keep the change on the existing bridge-driver/runtime seam by reusing the existing PeerTransportParametersCommitted update path rather than auto-committing or adding new broad handshake-completion state. - Keep PeerHandshakeTranscriptCompleted separate from the commit gate and prove that separation explicitly in tests. - Add focused requirement-home tests for no auto-commit before proof, positive commit after proof, repeated or conflicting commit rejection, fatal-state blocking, mismatched candidate rejection, runtime path consumption, and unchanged client-role semantics. - client identity, client-certificate authentication, trust-store policy, hostname validation, certificate-path validation, revocation, 0-RTT, 1-RTT, key update, endpoint-host wiring, harness changes, interop-runner handshake support, or any native TLS / QUIC fallback.  ## Verification Plan  Run the updated CRT and RFC9001 requirement-home tests that cover the explicit server-role post-proof gate, then run the relevant solution build and targeted test pass for the main library and tests project.  ## Completion Notes  The resulting slice is intentionally narrower than handshake completion.
