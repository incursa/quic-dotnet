# WI-QUIC-RFC9000-0021 - QUIC RFC 9000 CRYPTO Retransmission ACK Piggyback Work Item

## Summary

Implement and prove fresh pending ACK prefixing for rebuilt Initial and Handshake CRYPTO retransmission packets without treating old ACK bytes as retransmittable data.

## Requirements Addressed

- REQ-QUIC-RFC9000-S13P2P1-0009
- REQ-QUIC-RFC9000-S13P2P1-0010
- REQ-QUIC-RFC9000-S13P3-0006

## Design Inputs

- ARC-QUIC-RFC9000-0021

## Planned Changes

- Extend the long-header CRYPTO retransmission packet builders with a transient prefix-frame payload used for fresh ACK frame bytes.
- Wire the runtime CRYPTO retransmission rebuild path to prepend pending ACK frames for the matching Initial or Handshake packet number space.
- Mark piggybacked retransmission ACK frames as sent only after the rebuilt packet is tracked under its actual new packet number.
- Preserve existing behavior when no ACK is pending.
- Keep retry bootstrap replay, retried Initial replay, and broader delayed/path/interop scheduling outside this slice.
- Add focused positive, negative, deterministic fuzz/property, and BenchmarkDotNet coverage for the retransmission packet-builder path.

## Out of Scope

- Retry bootstrap replay ACK piggybacking.
- Retried Initial replay ACK piggybacking.
- Application Data stream and NEW_TOKEN ACK piggybacking already owned by earlier slices.
- A general delayed-ACK timer policy.
- Endpoint interop success classification.
- Public API behavior.
