## Planned Changes

- Preserve the bounded repeated-update control state that now exists above the first-successor floor for `REQ-QUIC-RFC9001-S6P5-0003`: current-phase acknowledgment tracking and the three-PTO post-confirmation local-update cooldown.
- Preserve the timing-neutral invalid-Key-Phase classification now proven for `REQ-QUIC-RFC9001-S6P3-0001` and the current timing-neutral no-close policy for apparent unconfirmed consecutive peer updates under `REQ-QUIC-RFC9001-S6P2-0003`.
- Preserve the helper-backed AEAD key lifecycle and policy floor now proven for `REQ-QUIC-RFC9001-S6P6-0001` through `REQ-QUIC-RFC9001-S6P6-0005`: per-key-set counters, confidentiality-limit stop-use, update-required classification, connection stop-use classification, and stateless-reset-only classification.
- Preserve the bounded runtime AEAD-limit wiring now added to 1-RTT send/open paths: first-update-before-excess-protection, successor-protection discard, and receive-side integrity-limit discard without close-datagram emission.
- Preserve the existing first-successor install path as a bounded prerequisite rather than widening its single-update guard in place.
- Keep repeated old/current/next epoch selection, second-successor packet handling, and broader stateless-reset-only response behavior blocked until those follow-ons have explicit requirements and proof plans.
- Keep TLS KeyUpdate prohibition, post-handshake ticket ingestion, 0-RTT, public API promises, and interop expansion out of this RFC 9001 lifecycle slice unless separately traced.

## Completion Notes

Planned for the broader repeated lifecycle. The direct first-successor proof for REQ-QUIC-RFC9001-S6P1-0001 through REQ-QUIC-RFC9001-S6P1-0010 is now executable at the requirement-home level, including retained old first-update 1-RTT packet-protection material before and after new-key packet authentication plus explicit retained cleanup. The first peer-initiated update proof now covers REQ-QUIC-RFC9001-S6P2-0001, REQ-QUIC-RFC9001-S6P2-0002, REQ-QUIC-RFC9001-S6P2-0003, and REQ-QUIC-RFC9001-S6P2-0004 for opening next-key packets only when they carry the next Key Phase bit, updating send keys before ACK emission, dropping apparent unconfirmed consecutive peer updates without close/datagram signaling under the current optional policy, and allowing KEY_UPDATE_ERROR when a retained old-key ACK packet acknowledges a tracked first-successor phase-1 send. The invalid Key Phase timing-neutral proof now covers REQ-QUIC-RFC9001-S6P3-0001 by dropping current/next key-phase mismatches without close/datagram signaling or successor installation. The first next-receive-key retention proof now covers REQ-QUIC-RFC9001-S6P3-0002 by retaining current and next receive material for apparent peer updates and clearing the retained next material once the first successor is installed. The first send-key ordering proof now covers REQ-QUIC-RFC9001-S6P4-0001 and REQ-QUIC-RFC9001-S6P4-0002 for first-update ACK sends and locally built higher-packet-number packets. The first old/current next receive-selection proof now covers REQ-QUIC-RFC9001-S6P4-0003, REQ-QUIC-RFC9001-S6P5-0001, and REQ-QUIC-RFC9001-S6P5-0002 for rejecting old-key packets that violate the first successor packet-number floor, using retained old receive material when recovered packet numbers fall below the current phase floor, and using next receive material when recovered packet numbers rise above that floor. The first repeated local-update cooldown proof now covers REQ-QUIC-RFC9001-S6P5-0003 by tracking acknowledgment of the current key phase and exposing a three-PTO post-confirmation eligibility gate without widening second-successor installation. The first retained-old discard proof now covers REQ-QUIC-RFC9001-S6P5-0004 and REQ-QUIC-RFC9001-S6P5-0005 by arming a three-PTO retained-old discard timer after the first authenticated phase-1 packet and discarding the retained old read material together with matching sender and recovery state when that timer expires. The helper-backed AEAD key lifecycle, policy, and bounded runtime proof now covers REQ-QUIC-RFC9001-S6P6-0001 through REQ-QUIC-RFC9001-S6P6-0005 for per-key-set protected/opened-packet counting, confidentiality-limit stop-use, update-required classification, connection stop-use classification, stateless-reset-only classification, first AEAD-limit-triggered key update on send, successor-protection discard, and receive-side integrity-limit discard without close-datagram emission. The remaining planned work is repeated epoch selection, second-successor packet handling, broader stateless-reset-only response behavior, receive-key selection beyond the first successor, and repeated old-key discard lifecycle; no general repeated 1-RTT key-update implementation is claimed by this artifact.
