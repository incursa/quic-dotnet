## Planned Changes

- Preserve the bounded repeated-update control state that now exists above the first-successor floor for `REQ-QUIC-RFC9001-S6P5-0003`: current-phase acknowledgment tracking and the three-PTO post-confirmation local-update cooldown.
- Preserve the bounded repeated local install floor that now derives and installs phase-2 material only after current-phase acknowledgment, three-PTO cooldown, and retained-old first-update discard, and emits outbound Key Phase using key-phase parity.
- Preserve the bounded repeated lower-recovered receive-selection floor that now authenticates retained phase-1 packets after local phase-2 install, rejects retained phase-1 packets carrying the phase-0 bit, and tracks the current packet-number floor per key phase.
- Preserve the timing-neutral invalid-Key-Phase classification now proven for `REQ-QUIC-RFC9001-S6P3-0001` and the current timing-neutral no-close policy for apparent unconfirmed consecutive peer updates under `REQ-QUIC-RFC9001-S6P2-0003`.
- Preserve the helper-backed AEAD key lifecycle and policy floor now proven for `REQ-QUIC-RFC9001-S6P6-0001` through `REQ-QUIC-RFC9001-S6P6-0005`: per-key-set counters, confidentiality-limit stop-use, update-required classification, connection stop-use classification, and stateless-reset-only classification.
- Preserve the bounded runtime AEAD-limit wiring now added to 1-RTT send/open paths: first-update-before-excess-protection, successor-protection discard, and receive-side integrity-limit discard without close-datagram emission.
- Preserve the existing first-successor install path as a bounded prerequisite rather than widening receive-side repeated epoch selection in place.
- Keep peer-initiated repeated updates, next-key second-successor packet acceptance before local install, repeated AEAD-limit-triggered updates, and broader stateless-reset-only response behavior blocked until those follow-ons have explicit requirements and proof plans.
- Keep TLS KeyUpdate prohibition, post-handshake ticket ingestion, 0-RTT, public API promises, and interop expansion out of this RFC 9001 lifecycle slice unless separately traced.

## Traffic Secret Advancement Slice

- Add a pending 1-RTT traffic-secret update carrier so derived successor secrets are zeroed unless the matching bridge-state install commits.
- Replace stored application traffic secrets with the updated `quic ku` secrets when the first local or peer-initiated update commits.
- Allow helper derivation of the next successor material from the advanced secrets while keeping repeated update installation and second-successor packet acceptance blocked.

## Bounded Repeated Local Install Slice

- Install a repeated local update only after current-phase acknowledgment, three-PTO cooldown, and retained-old first-update discard.
- Retain the displaced phase-1 open/protect material when phase 2 becomes current.
- Emit outbound Key Phase using key-phase parity so phase 2 writes bit 0.
- Keep repeated receive epoch selection, second-successor packet acceptance, peer-initiated repeated updates, repeated AEAD-limit-triggered updates, and broader stateless-reset-only response behavior blocked.

## Bounded Repeated Lower-Recovered Receive Slice

- Authenticate lower recovered phase-1 packets with retained previous material after bounded local phase-2 install.
- Reject retained phase-1 packets that carry the phase-0 bit after phase 2 becomes current.
- Track the current packet-number floor by key-phase counter so first-update phase-1 observations do not poison phase-2 ordering.
- Keep peer-initiated repeated updates and next-key second-successor packet acceptance before local install blocked.

## Next Planned Scope

- Design and prove peer-initiated repeated update handling plus next-key second-successor packet acceptance now that bounded local phase-2 install and lower-recovered retained phase-1 selection are implemented.
- Keep repeated AEAD-limit-triggered updates and broader stateless-reset-only response behavior blocked until their requirements are traced.
