## Planned Changes

- Preserve the bounded repeated-update control state that now exists above the first-successor floor for `REQ-QUIC-RFC9001-S6P5-0003`: current-phase acknowledgment tracking and the three-PTO post-confirmation local-update cooldown.
- Preserve the timing-neutral invalid-Key-Phase classification now proven for `REQ-QUIC-RFC9001-S6P3-0001` and the current timing-neutral no-close policy for apparent unconfirmed consecutive peer updates under `REQ-QUIC-RFC9001-S6P2-0003`.
- Preserve the helper-backed AEAD key lifecycle and policy floor now proven for `REQ-QUIC-RFC9001-S6P6-0001` through `REQ-QUIC-RFC9001-S6P6-0005`: per-key-set counters, confidentiality-limit stop-use, update-required classification, connection stop-use classification, and stateless-reset-only classification.
- Wire those helper-backed AEAD-limit decisions into runtime 1-RTT send/open paths through the same repeated-update gate used for ordinary local initiation.
- Preserve the existing first-successor install path as a bounded prerequisite rather than widening its single-update guard in place.
- Keep repeated old/current/next epoch selection and second-successor packet handling blocked until runtime AEAD-policy wiring is proven.
- Keep TLS KeyUpdate prohibition, post-handshake ticket ingestion, 0-RTT, public API promises, and interop expansion out of this RFC 9001 lifecycle slice unless separately traced.

Planned for the broader repeated lifecycle. The first peer-initiated update proof now covers REQ-QUIC-RFC9001-S6P2-0001, REQ-QUIC-RFC9001-S6P2-0002, REQ-QUIC-RFC9001-S6P2-0003, and REQ-QUIC-RFC9001-S6P2-0004 for opening next-key packets only when they carry the next Key Phase bit, updating send keys before ACK emission, dropping apparent unconfirmed consecutive peer updates without close/datagram signaling under the current optional policy, and allowing KEY_UPDATE_ERROR when a retained old-key ACK packet acknowledges a tracked first-successor phase-1 send. The invalid Key Phase timing-neutral proof now covers REQ-QUIC-RFC9001-S6P3-0001 by dropping current/next key-phase mismatches without close/datagram signaling or successor installation. The first next-receive-key retention proof now covers REQ-QUIC-RFC9001-S6P3-0002 by retaining current and next receive material for apparent peer updates and clearing the retained next material once the first successor is installed. The first send-key ordering proof now covers REQ-QUIC-RFC9001-S6P4-0001 and REQ-QUIC-RFC9001-S6P4-0002 for first-update ACK sends and locally built higher-packet-number packets. The first old/current next receive-selection proof now covers REQ-QUIC-RFC9001-S6P4-0003, REQ-QUIC-RFC9001-S6P5-0001, and REQ-QUIC-RFC9001-S6P5-0002 for rejecting old-key packets that violate the first successor packet-number floor, using retained old receive material when recovered packet numbers fall below the current phase floor, and using next receive material when recovered packet numbers rise above that floor. The first repeated local-update cooldown proof now covers REQ-QUIC-RFC9001-S6P5-0003 by tracking acknowledgment of the current key phase and exposing a three-PTO post-confirmation eligibility gate without widening second-successor installation. The first retained-old discard proof now covers REQ-QUIC-RFC9001-S6P5-0004 and REQ-QUIC-RFC9001-S6P5-0005 by arming a three-PTO retained-old discard timer after the first authenticated phase-1 packet and discarding the retained old read material together with matching sender and recovery state when that timer expires. The helper-backed AEAD key lifecycle and policy proof now covers REQ-QUIC-RFC9001-S6P6-0001 through REQ-QUIC-RFC9001-S6P6-0005 for per-key-set protected/opened-packet counting, confidentiality-limit stop-use, update-required classification, connection stop-use classification, and stateless-reset-only classification without claiming runtime send/open path wiring for those policy decisions. The remaining planned work is runtime wiring for AEAD-limit decisions; no repeated 1-RTT key-update implementation, receive-key selection beyond the first successor, runtime AEAD-limit update-or-close application, or repeated old-key discard lifecycle is claimed by this artifact before that wiring lands.
