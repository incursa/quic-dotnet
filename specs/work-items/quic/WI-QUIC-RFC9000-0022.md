# WI-QUIC-RFC9000-0022 - QUIC RFC 9000 Retry Replay ACK Piggyback Work Item

## Summary

Implement and prove fresh pending Initial ACK prefixing for Retry-selected Initial replay packets without changing Retry token echoing, Retry-selected destination connection ID, retained ClientHello bytes, or packet-number continuity.

## Planned Changes

- Wire the Retry-selected Initial replay helper to build a transient Initial ACK frame payload when Initial ACK-generation state has pending ack-eliciting receipts.
- Pass that transient ACK payload to the existing Initial packet builder before the replayed CRYPTO frame while preserving the Retry token and Retry-selected destination connection ID.
- Mark the piggybacked Initial ACK frame as sent only after the replay packet is tracked under its actual new Initial packet number.
- Preserve existing behavior when no Initial ACK is pending.
- Keep ordinary non-Retry bootstrap probe replay, delayed-ACK scheduling, path scheduling, interop success, and public API behavior outside this slice.
- Add focused positive, negative, deterministic fuzz/property, and BenchmarkDotNet coverage for the Retry-selected Initial replay packet-builder path.

## Out of Scope

- Ordinary non-Retry bootstrap probe replay ACK piggybacking.
- Application Data stream, NEW_TOKEN, live long-header CRYPTO, and CRYPTO retransmission ACK piggybacking already owned by earlier slices.
- A general delayed-ACK timer policy.
- Candidate-path or multipath scheduling.
- Endpoint interop success classification.
- Public API behavior.
