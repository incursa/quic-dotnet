# WI-QUIC-RFC9000-0024 - QUIC RFC 9000 Application Data Delayed ACK Timer Work Item

## Summary

Implement and prove the active-path Application Data ACK-delay runtime timer so ordinary ACK-only packets wait for local max_ack_delay while immediate ACK exceptions, packet-count threshold sends, and ACK piggybacking still work.

## Planned Changes

- Add a dedicated runtime ACK-delay timer kind and deadline state.
- Arm the timer after ordinary 1-RTT ack-eliciting packet processing when no immediate ACK or packet-count threshold send is required.
- Emit an ACK-only packet at timer expiry without injecting ack-eliciting frames.
- Keep non-ack-eliciting-only receipts from arming the timer.
- Cancel the timer after ACK-only sends or ACK piggybacking on outgoing ack-eliciting packets.
- Preserve Initial, Handshake, path scheduling, interop success, and public API behavior outside this slice.
- Add positive, negative, edge, deterministic fuzz/property, and BenchmarkDotNet coverage for the timer decision path.

## Out of Scope

- Initial and Handshake ACK-only scheduling.
- Candidate-path or multipath ACK scheduling.
- Endpoint interop success classification.
- Public API behavior.
- Transport-parameter public option redesign.
- ACK frame parser or serializer behavior.
