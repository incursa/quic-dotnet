# QUIC Requirement Gaps

This ledger tracks open questions, ambiguities, and follow-up decisions for QUIC requirement work.

## Open Gaps
- `public-api-stream-entry-slice` is now the active phase boundary for `SPEC-QUIC-API`. The repository has already promoted the client-entry consumer surface (`QuicClientConnectionOptions` and `QuicConnection.ConnectAsync(...)`) on top of the existing connection/runtime shell and the supported positive loopback establishment boundary, and the current narrow slice promotes `QuicConnection.AcceptInboundStreamAsync(...)`, `QuicConnection.OpenOutboundStreamAsync(...)`, `QuicStreamCapacityChangedArgs`, `QuicConnectionOptions.StreamCapacityCallback`, the supported `QuicStream` write/completion lane, and the narrow `RESET_STREAM` / `STOP_SENDING` abort pair on top of the same runtime/stream-state seams with only the minimum real 1-RTT short-header stream-control path needed for a truthful supported loopback stream open/accept/write/completion/abort flow plus initial peer-advertised outbound stream-capacity reporting, later real peer `MAX_STREAMS` growth on the supported active loopback path, and the narrow supported close-driven capacity-release subset on that same path. Combined `Abort(Both, ...)`, broader stream-management parity, 0-RTT, key update, and interop-runner enablement remain deferred.
- `interop-harness` now has a thin endpoint-host shell around the library runtime, and the managed client/listener host path already owns honest Initial/DCID bootstrap plus server Initial-response emission; the harness `handshake` testcase now dispatches into that managed path, the narrow transfer testcase now dispatches through the managed active-phase path, and the child-process-only `retry` testcase now dispatches through the managed one-Retry path under `REQ-QUIC-INT-0012`. The main library remains the owner of the connection runtime, TLS-facing transport bridge, sender/recovery ownership, and diagnostics hooks.
- `9001-05-handshake-orchestration` is still incomplete overall. The library now has a transport-facing TLS bridge state under `REQ-QUIC-CRT-0103`, a narrow Handshake flow coordinator under `REQ-QUIC-CRT-0106`, a role-aware transcript-progress owner under `REQ-QUIC-CRT-0107`, a managed client-only TLS 1.3 proof slice under `REQ-QUIC-CRT-0108` and `REQ-QUIC-CRT-0110`, a separate local certificate-acceptance seam under `REQ-QUIC-CRT-0111`, a server-role managed TLS crypto floor under `REQ-QUIC-CRT-0112`, a follow-on server-role local `EncryptedExtensions` continuation under `REQ-QUIC-CRT-0113`, a narrow server-role local `Certificate` continuation under `REQ-QUIC-CRT-0114`, a narrow server-role local `CertificateVerify` continuation under `REQ-QUIC-CRT-0115`, a narrow server-role local `Finished` continuation under `REQ-QUIC-CRT-0116`, a server-role inbound client `Finished` proof slice under `REQ-QUIC-CRT-0117`, a narrow server-role peer transport-parameter commit gate under `REQ-QUIC-CRT-0118`, a server-role 1-RTT packet-protection publication slice under `REQ-QUIC-CRT-0119`, and the client-role 1-RTT readiness prerequisite under `REQ-QUIC-CRT-0121`. That combined handshake slice is now green in the current implementation and remains intentionally narrow: it is limited to TLS 1.3 with `TLS_AES_128_GCM_SHA256` and `secp256r1`; the client role derives handshake secrets from the local ephemeral share, peer `ServerHello` key share, and transcript hash, verifies the peer leaf `CertificateVerify` proof with the supported algorithm subset, verifies peer `Finished` cryptographically, accepts only an explicit pinned leaf SHA-256 policy, and gates peer transport-parameter commit on both proof states plus local acceptance; the server role accepts a supported peer `ClientHello`, constructs a local `ServerHello`, derives Handshake packet-protection material from the resulting transcript boundary, emits a local `EncryptedExtensions` using the local transport parameters on the existing handshake seam, emits a narrow local `Certificate` from an explicit raw-DER leaf certificate seam on the same handshake seam, emits one local `CertificateVerify` from an explicit raw signing-key seam compatible with that leaf certificate, emits one local `Finished` from the same managed handshake secret and transcript boundary, verifies the inbound client `Finished` against the complete local server flight, surfaces outbound `ServerHello` plus `EncryptedExtensions` plus `Certificate` plus `CertificateVerify` plus `Finished` bytes through the existing bridge seam, publishes `PeerFinishedVerified` plus the narrow `PeerHandshakeTranscriptCompleted` milestone after the proof succeeds, allows the server-role peer transport-parameter commit gate to open only after that proof boundary, and separately publishes the first server-role 1-RTT open/protect packet-protection material only after the same supported proof boundary. Remaining adjacent blockers are certificate trust-store policy, hostname and identity policy, certificate-path validation, the first client-side 0-RTT packet-emission attempt floor, key update, Initial/DCID bootstrap, and interop-runner handshake enablement.
- `9002-06-key-discard-lifecycle` is partially reusable at the helper layer. The repo now has an explicit TLS bridge and connection-runtime discard path for Initial/Handshake key cleanup, but the RFC 9002 S6P4 0-RTT rejection and secret-discard timing clauses remain blocked until the repo has TLS handshake orchestration, 0-RTT state, and full 1-RTT key-update mechanics.
- `9002-06-appendix-b-constants-and-examples` is split into a helper-backed executable subset and a deferred remainder. The helper-backed `SBP1`, `SBP2-0002`, `SBP2-0004`, `SBP2-0005`, `SBP3`, `SBP4`, `SBP5`, `SBP6`, `SBP7`, and `SBP8` clauses can move now, while `REQ-QUIC-RFC9002-SBP2-0001`, `REQ-QUIC-RFC9002-SBP2-0003`, and `REQ-QUIC-RFC9002-SBP9-0001` through `REQ-QUIC-RFC9002-SBP9-0003` remain blocked until the repo has sender/runtime PMTU accounting and connection-owned key-discard cleanup. The retained `REQ-QUIC-RFC9002-SBP9-0003` / `REQ-QUIC-RFC9002-SAP11-0003` overlap stays under manual review.
- `9000-19-retransmission-and-frame-reliability` is split across send-path, recovery, stream-lifecycle, flow-control, path-validation, connection-ID, and congestion-control behaviors that are not present in the current helper-only transport slice. The helper-backed subset now closes `REQ-QUIC-RFC9000-S13P3-0010` and `REQ-QUIC-RFC9000-S13P3-0027`; 25 clauses remain partial and 12 remain blocked until the repo grows sender/recovery orchestration and the other missing transport surfaces.
- `9000-02-stream-state` is partially reusable at the helper layer: 16 clauses across S3/S3P1/S3P2 are now proven by helper-state tests, while 10 clauses remain partially implemented and 40 remain blocked until the repo grows the application-facing stream abstraction, sender/recovery orchestration, and broader abort coordination.
- `9000-03-flow-control` is substantially reusable at the helper layer: 43 clauses across S4/S4P1/S4P2/S4P4/S4P5/S4P6 are now proven by helper-state, codec, and transport-parameter tests. The remaining explicit blockers are `REQ-QUIC-RFC9000-S4P5-0001` (partial until sender/recovery reliability exists), `REQ-QUIC-RFC9000-S4P1-0015`, `REQ-QUIC-RFC9000-S4P2-0001` through `REQ-QUIC-RFC9000-S4P2-0004`, and `REQ-QUIC-RFC9000-S4P6-0013`.
- `9000-11-migration-core` is mostly blocked by the absence of the broader path-selection, packet-sending, and ECN/timer surfaces. The helper-backed anti-amplification slice now closes `REQ-QUIC-RFC9000-S9P3P1-0001`, and the runtime path-state/migration orchestration slice now covers active/candidate path ownership, address-change classification, and promotion rules, but the remaining S9/S9P1/S9P2/S9P3 requirements still need a richer sender and ECN surface before they can be closed cleanly.
- `9000-13-idle-and-close` is partially reusable at the helper layer, and the repo now has a connection close/drain lifecycle runtime slice that closes the helper-backed no-send clauses for `REQ-QUIC-RFC9000-S10P2P2-0001` and `REQ-QUIC-RFC9000-S10P2P2-0003`. Idle-timeout floor and restart bookkeeping remain helper-backed, while the remaining RFC 9000 lifecycle clauses now route through the runtime seam and the CRT lifecycle requirements.
- `9000-14-stateless-reset` is partially reusable at the helper layer. The repo now closes the helper-backed packet-layout, token-generation, amplification-sizing, 38-bit visible-prefix, and matched-token drain/no-send clauses, but the explicit endpoint blockers remain `REQ-QUIC-RFC9000-S10P3-0001`, `REQ-QUIC-RFC9000-S10P3-0015`, token-retirement invalidation, token memory scoping by remote address, long-header reset policy, version-aware reset acceptance, and reset-send limiting, all of which still need the endpoint receive/send lifecycle surface.
- `9001-02-security-and-registry` is partially reusable at the helper layer. The helper-backed ceiling for the chunk closes `REQ-QUIC-RFC9001-S6-0002`, `REQ-QUIC-RFC9001-S8-0001`, and `REQ-QUIC-RFC9001-S10-0001` through `REQ-QUIC-RFC9001-S10-0003`. The bridge now has executable seam coverage for carrying and authenticating transport parameters plus routing TLS KeyUpdate violations into explicit connection-close handling, but the remaining security clauses still depend on a real TLS core and packet-protection engine. `REQ-QUIC-RFC9001-S7-0001` and `REQ-QUIC-RFC9001-S9-0001` remain deferred because they are policy/document-scope items rather than executable helper behavior.

## Closed Gaps

- `host-facing-qlog-capture-serialization` is closed. The narrow host-facing capture helper above `Incursa.Quic.Qlog` now collects per-connection traces in memory and serializes them through the existing `Incursa.Qlog.Serialization.Json` path without moving file policy into `Incursa.Quic`. The remaining trace now lives under `WI-QUIC-CRT-0035`, `REQ-QUIC-CRT-0138`, and `VER-QUIC-CRT-0035`.
- `diagnostics-qlog-adapter-boundary` is closed. The sibling `Incursa.Quic.Qlog` adapter now maps the typed transport diagnostics emitted by `Incursa.Quic` into `Incursa.Qlog.Quic` without moving serializer or file ownership into the transport core. The remaining trace now lives under `WI-QUIC-CRT-0033`, `REQ-QUIC-CRT-0136`, and `VER-QUIC-CRT-0033`.
- `post-handshake-ticket-update-seam` is closed. The library now has the narrow internal post-Finished ticket-bearing TLS update seam tracked separately as `REQ-QUIC-CRT-0128`, `ARC-QUIC-CRT-0026`, `WI-QUIC-CRT-0026`, and `VER-QUIC-CRT-0026`; it surfaces opaque ticket bytes internally after Finished without ticket persistence, replay ownership, PSK derivation, resumed-handshake support, 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening. This remains distinct from the real client-side 1-RTT wire-ingress follow-on under `REQ-QUIC-CRT-0129`.
- `post-handshake-ticket-wire-ingress` is closed. The client-side 1-RTT `NewSessionTicket` ingress follow-on is tracked separately as `REQ-QUIC-CRT-0129`, `ARC-QUIC-CRT-0027`, `WI-QUIC-CRT-0027`, and `VER-QUIC-CRT-0027`; it consumes a valid TLS 1.3 `NewSessionTicket` after Finished, extracts opaque ticket bytes through the existing internal seam from `REQ-QUIC-CRT-0128`, keeps pre-Finished and server-role ingress rejected, and keeps unsupported 1-RTT post-handshake TLS messages ignored rather than broadening the parser into a general post-handshake engine. It does not claim ticket persistence, replay ownership, PSK derivation, resumed-handshake support, 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `0rtt-resumption-ticket-ownership` is closed. The managed client/runtime path now owns an opaque resumption-ticket snapshot distinct from the transient bridge-state ingress store and keeps the early-data gate explicitly closed, tracked under `REQ-QUIC-CRT-0127`, `ARC-QUIC-CRT-0025`, `WI-QUIC-CRT-0025`, and `VER-QUIC-CRT-0025`. This slice does not claim PSK derivation, replay handoff, 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `detached-resumption-ticket-handoff` is closed. The managed client/runtime path now exports an opaque detached resumption-ticket carrier from the owned current-lifetime snapshot and accepts that carrier as dormant internal state in a later client setup, tracked under `REQ-QUIC-CRT-0130`, `ARC-QUIC-CRT-0028`, `WI-QUIC-CRT-0028`, and `VER-QUIC-CRT-0028`. This slice stays distinct from transient bridge-state ingress storage and the current runtime-owned snapshot, and it does not claim PSK derivation, ClientHello replay, resumed-handshake support, 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `detached-resumption-credential-material-capture` is closed. The managed client/runtime path now retains the minimum internal resumption-credential material needed for a later honest attempt, tracked under `REQ-QUIC-CRT-0131`, `ARC-QUIC-CRT-0029`, `WI-QUIC-CRT-0029`, and `VER-QUIC-CRT-0029`. This closed slice remains client-role only, keeps early data explicitly closed, and does not claim ClientHello PSK-attempt success or resumed-handshake success.
- `clienthello-psk-attempt-path` is closed. The managed client/runtime path now consumes the dormant detached carrier to construct a PSK-capable ClientHello attempt with binder material, tracked under `REQ-QUIC-CRT-0132`, `ARC-QUIC-CRT-0030`, `WI-QUIC-CRT-0030`, and `VER-QUIC-CRT-0030`. This slice remains client-role only, keeps early data explicitly closed, and does not claim resumed-handshake success, 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `serverhello-resumption-branch-point` is closed. The managed client/runtime path now classifies a PSK-capable ClientHello attempt at ServerHello as accepted or rejected, tracked under `REQ-QUIC-CRT-0133`, `ARC-QUIC-CRT-0031`, `WI-QUIC-CRT-0031`, and `VER-QUIC-CRT-0031`. Rejected attempts fall back cleanly to the existing full-handshake path without corrupting dormant client state, accepted attempts are recognized but do not claim resumed-handshake success, and early data remains explicitly closed. This slice stays client-role only and does not claim 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `client-side-abbreviated-resumption-completion` is closed. The managed client/runtime path now advances an accepted PSK-capable ClientHello attempt through the abbreviated `EncryptedExtensions -> Finished` resumption flight, tracked under `REQ-QUIC-CRT-0137`, `ARC-QUIC-CRT-0033`, `WI-QUIC-CRT-0034`, and `VER-QUIC-CRT-0034`. Rejected attempts continue to fall back cleanly to the existing full-handshake path, early-data admission remains explicitly closed, and the slice does not claim 0-RTT packet support, anti-replay, key update, or any public API widening.
- `early-data-prerequisite-capture` is closed. The managed client/runtime path now carries the minimum dormant early-data prerequisite material behind the detached resumption carrier: a positive `max_early_data_size` hint from `NewSessionTicket` plus a cloned peer transport-parameter snapshot from the prior connection, tracked under `REQ-QUIC-CRT-0139`, `ARC-QUIC-CRT-0034`, `WI-QUIC-CRT-0036`, and `VER-QUIC-CRT-0036`. Later managed client setup retains that richer carrier as dormant internal state, early-data admission stays explicitly closed, and the slice does not claim 0-RTT packet support, anti-replay, key update, transfer, retry, or any public API widening.
- `dormant-early-data-attempt-readiness` is closed. The managed client/runtime path now exposes a narrow internal readiness observation over the dormant detached resumption carrier when it contains both resumption credential material and the early-data prerequisite material, tracked under `REQ-QUIC-CRT-0140`, `ARC-QUIC-CRT-0035`, `WI-QUIC-CRT-0037`, and `VER-QUIC-CRT-0037`. Later managed client setup retains the dormant carrier, early-data admission stays explicitly closed, and the slice does not claim 0-RTT packet send/receive, anti-replay, key update, transfer, retry, or any public API widening.
- `client-0rtt-packet-emission-attempt-floor` is closed. The managed client/runtime path now admits the first bounded 0-RTT send-attempt from the dormant detached resumption carrier when both the resumption credential material and the early-data prerequisite material are present, derives distinct 0-RTT packet-protection material for the first protected client application packet, and falls back cleanly when the carrier or derived material is absent. The remaining trace now lives under `REQ-QUIC-CRT-0141`, `ARC-QUIC-CRT-0036`, `WI-QUIC-CRT-0038`, and `VER-QUIC-CRT-0038`.
- `server-client-auth-floor` is closed. The managed server path now honors `QuicServerConnectionOptions.ServerAuthenticationOptions.ClientCertificateRequired` on the existing carrier, emits `CertificateRequest`, accepts inbound client `Certificate` / `CertificateVerify` / `Finished` through the existing callback seam, and keeps the server-side `CertificateChainPolicy` and standalone `CertificateRevocationCheckMode` follow-ons on separate slices while broader PKI, 0-RTT, key update, transfer, retry, and any public API widening beyond the existing carrier remain out of scope.
- `server-client-auth-chain-policy-follow-on` is closed. The managed server path now honors `SslServerAuthenticationOptions.CertificateChainPolicy` on the callback-driven `ClientCertificateRequired` path, delegates presented-client-certificate evaluation to BCL `X509Chain` / `X509ChainPolicy` primitives before invoking the existing callback seam, and fails closed on trust/policy mismatch without widening standalone `CertificateRevocationCheckMode` interaction, broader PKI/path-building, 0-RTT, key update, transfer, or retry behavior.
- `server-client-auth-revocation-mode-follow-on` is closed. The managed server path now honors `SslServerAuthenticationOptions.CertificateRevocationCheckMode` on the callback-driven `ClientCertificateRequired` path when `CertificateChainPolicy` is absent, delegates revocation configuration to BCL `X509Chain` / `X509ChainPolicy` primitives before invoking the existing callback seam, and rejects mixed `CertificateChainPolicy` plus standalone revocation configuration without widening broader PKI/path-building, 0-RTT, key update, transfer, or retry behavior.
- `client-trust-policy-follow-on` is closed. The managed client path now accepts `QuicClientConnectionOptions.PeerCertificatePolicy` carrying `QuicPeerCertificatePolicy.ExactPeerLeafCertificateDer` and `QuicPeerCertificatePolicy.ExplicitTrustMaterialSha256`, feeds those bytes into the internal snapshot under `REQ-QUIC-API-0012` and `REQ-QUIC-CRT-0123`, and also supports the mainstream BCL-shaped `SslClientAuthenticationOptions` validation path for `TargetHost`, `CertificateChainPolicy`, callback overrides, and revocation delegation. Keep broader client-auth, `0-RTT`, key update, transfer, and retry out of scope.
- `public-validation-standard-client-tls-mode` is closed. The mainstream client path now accepts `TargetHost`, `CertificateChainPolicy`, callback overrides, and revocation delegation on the existing `SslClientAuthenticationOptions` carrier, while `QuicPeerCertificatePolicy` remains the separate exact-pinning floor. Keep broader client-auth, `0-RTT`, key update, transfer, and retry out of scope.
- `client-trust-policy-follow-on` is closed. The managed client path now accepts `QuicClientConnectionOptions.PeerCertificatePolicy` carrying `QuicPeerCertificatePolicy.ExactPeerLeafCertificateDer` and `QuicPeerCertificatePolicy.ExplicitTrustMaterialSha256`, feeds those bytes into the internal snapshot under `REQ-QUIC-API-0012` and `REQ-QUIC-CRT-0123`, and also supports the mainstream BCL-shaped `SslClientAuthenticationOptions` validation path for `TargetHost`, `CertificateChainPolicy`, callback overrides, and revocation delegation. Keep broader client-auth, `0-RTT`, key update, transfer, and retry out of scope.
